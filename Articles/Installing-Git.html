Installing, Configuring, and using Git

The beginners linux user's guide to the most popular version control system.

Prerequesites

For this tutorial you will need:

  • A Linux distro
  • Terminal access

It is also recommended that you have:

  • Root access (if git isn't installed)
  • A github account and repository
  • An internet connection

What is git?

Git is an extremely popular, free, open source, distributed version control system (DVCS). This means it will keep track of code being added, updated, or modified in any way. The Distributed part means that git can use every copy of the code as a repository to store the full history of the project. Even if all copies on every server, backup server, backup backup server are destroyed, if a developer is using git to work on it then it can be restored.

However, git is not the only DVCS, so why is it so popular? Git has been actively maintained since 2005, where it was created by none other than Linus Torvalds, the creator of Linux himself. Git has been tried and tested for more than 15 years, and it is one of the fastest, lightest and most documented programs of it's kind. Git is now a key part of so many sytems and development teams, even this website was written and deployed to my web server using git.

Seeing if we have git installed

Note: Returning back again? Have a look at the Cheatsheet

Most decently sized desktop versions of linux already come with git pre-installed, as it is such a widely used program. You can check very simply by running the command:
git --version

Which should hopefully return a message telling you your current version of git:
git version 2.23.4

If this runs correctly you have git installed already and you can skip straight to Configuring git, but don't panic if you recieve an error saying "git: command not found", this just means you dont have git preinstalled and you need to simply install it...

Installing Git on Linux

Installing any popular application is very straightforward on linux, through the use of a package manager. simply type or even copy and paste the command below that corresponds to your package manager or your distro into the terminal.

For Ubuntu and Ubuntu based distributions:
sudo apt install git

For Arch and Arch based distributions:
sudo pacman -S git

For Gentoo and Gentoo based distributions:
sudo emerge git

For Fedora and other Red Hat based distributions:
sudo yum install git

Now run again the git --version command and you should see that git is successfully installed on your system. You can now configure it. Still unsure? Try googling what package manager your distro uses and how to install packages with it.

Configuring Git

Now that we have git installed, we need to configure a few variables in order to get full use out of it. The main things we need to set the name and email Git will tie our commits to. Use these commands, replacing Name and Email with valid information that should would link to any github account you plan to use.

git config --global user.name “Name”
git config --global user.email "Email"

Optionally, we can also configure ssh keys for git and github, allowing for commits without a password. I have made an article on it here but we will skip it for this tutorial as it is not essential and is a more complex process for someone who is just starting out with git, so for now we will move on with learning how to use git

Understanding git

In order to use git, we need to understand more of how git works. There are a couple of terms you should know, as they are crucial to a proper understanding of git. Firstly, we have a Repository, commonly called a Repo. Think of this like a master folder in which all of your git enabled project should be stored.

Next we have a Branch. This is like, well a branch off of your main project. All new repositories start with a Master/Main branch, and more can be created easily. Branches are used so we can organize code into for example: a stable branch that would be currently deployed, a Main branch where people working on the project can commit their newest changes, feature branches for new features and hotfix branches for quick bug fixes. Branches, and git in general are all about being more organized, making development easier, especially on a larger scale team.

Pretty straightforward so far right? well now we understand how git organizes code, we need to learn how to 'upload' it. Firstly a file must be registered for commitment, then commited with a message describing what we have changed, then it can be pushed from our local repository to the remote one, like your github repository for example.

For now this is all you need to know about git, time to Build our first repository

Creating our first repository

Note: If you don't want to create your own repository, you can Clone an existing Repository

In order to create our first repository, we first need a directory for it, lets create one using this command:
mkdir myFirstRepository

Then we can then Change into this directory using:
cd myFirstRepository

Now we can finally use our first real git command: git init. this will initialize our repository, creating the .git directory and allowing us to use git with it.
git init

Now lets create two basic files:
touch myFileOne
touch myFileTwo

To 'register' these files for commitment we can use the commands:
git add myFileOne
git add myfileTwo

But seeing as we want to add all the files anyway we can use this command to add all modified files and directories to the commit list:
git add -A

Now to commit these files with a message describing what we have changed we can do:
git commit -m "Upload Files"

Now our changes have been commited to our local repository on the main branch, but what if we want to commit it to a remote server like github to access anywhere? This is very simple, we just add the url of our github repo, then push our commits to it using the following two commands:
git remote add myFirstRemoteRepo https://github.com/username/repository
git push myFirstRemoteRepo

Well done, you have just created, modified and commited to your first repository, and commited it to a remote server. You have only a few more basic commands to learn and you will have fully mastered git!

Common Practices and additional commands

Although sometimes creating a local repo first is the only option, I personally always create new repositories on github first, and clone them to my computer. This is also how you would clone code to another machine that doesnt have any of the code downloaded.
git clone https://github.com/username/repository

If you have more than one developer working on a remote repository or have commited code on several different machines, you can have issues where code has been altered but is out of date, even if you are working alone, it is a best practise to pull from the remote repo before you push your new code.
git pull remoteName branch

If however, pushing still returns an error stating some refs could not be pushed to remote, then you may need to rebase the branch using:
git pull --rebase remoteName branch

To manage branches you can use the following three commands to list branches, create new branches, and switch to another branch.
git branch
git branch branchName
git checkout branchName

Git Command Cheatsheet

I don't know about you, but I love Cheatsheets. This section will probably see the most use out of me. Been on holiday for a week and forgot some commands? cheatsheet. Want to use that one obscure command you skipped reading up about? cheatsheet. been a long time gui user but dont know any git cli commands? cheatsheet! If you are here because you've read this entire page and are ready to start learning git, bookmark this page and you will learn every command just by visiting here each time you forget. Major credit to github for having such great documentation and accurate but short descriptions of every command listed below:

Configuration

set a name that is identifiable for credit when review version history
git config --global user.name "name"
set an email address that will be associated with each history marker
git config --global user.email "email"

Cloning & Init

initialize an existing directory as a Git repository
git init
retrieve an entire repository from a hosted location via URL
git clone https://github.com/username/repository

Staging

show modified files in working directory, staged for your next commit
git status
add a file in its current save to your next commit
git add fileName
unstage a file while retaining the changes in working directory
git reset fileName
commit your staged content as a new commit with a message
git commit -m "Meaningful Message"

Branches

list your branches. a * will appear next to the currently active branch
git branch
create a new branch at the current commit
git branch branchName
switch to another branch and check it out into your working directory
git checkout branchNamemerge the specified branch into the current one
git merge branchName

Share & Update

add a git url as an alias
git remote add remoteRepoName https://github.com/username/repository
push local commits to remote repository branch
git push remoteRepoName branchName
fetch and merge any commits from the tracking remote branch
git pull
apply any commits of current branch ahead of specified one
git rebase branchName
